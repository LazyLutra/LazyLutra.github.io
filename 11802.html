Bean的加载和控制

发表于2022-05-02|更新于2022-05-17|SpringBoot

Bean的加载方式

第一种：XML

使用该方式声明 Bean 时，Spring 会调用 Bean 中的无参构造方法。

第二种：注解

  1. 使用 @Component 及其衍生注解，@Controller、@Service、@Repository 定义Bean。
  2. 使用 @Bean 定义第三方 Bean，并将其所在类定义为配置类或 Bean，@Configuration。

第三种：配置类和FactoryBean

配置类

  1. 约等于第二种方法，把 XML 配置文件的形式变成配置类。

  2. 使用 @ImportResource 注解加载配置类并且加载其他的配置文件

FactoryBean

第四种：使用@Import注解

被导入的 Bean 不需要使用注解声明为 Bean，可以有效降低代码与 Spring 的耦合。

输出的结果为全路径类名：com.bean.Dog。

也可以导入配置类，且被导入的配置类不需要使用 @Configuration 注解，配置类下声明的其他类也会被注入。

第五种：上下文容器对象初始化完成后注入

⭐| 第六种：实现ImportSelector接口

  1. 使用了@Configuration 注解
  2. 没有使用@Configuration 注解

第七种：实现ImportBeanDefinitionRegistrar接口

通过 BeanDefinition 注册器注册实名 Bean，实现对容器中 Bean 的处理，例如：覆盖现有 Bean。

第八种：后置处理器 PostProcessor

最后处理 Bean 的注册，可以保障兜底。

Bean的加载控制

第五种 ~ 第八种 都支持根据条件加载 Bean，此处使用第五种方式做示例。

注解控制（Condition）

Spring 4.0 新增的条件判断功能，通过该功能可以实现选择性的创建 Bean 操作。

用于判断当前环境中是否导入相关类。

问题：如何判断 Bean 注入成功？

SpringBoot 自带的 Condition

路径为：spring-boot-autoconfigure.jar \org\springframework\boot\autoconfigure\condition

小结

  • 自定义条件流程

    1. 创建类实现 Condition 接口并重写 matches 方法，并且在 matches 方法中进行判断，返回 boolean 值。
      • matches 方法有两个参数：
        1. context：上下文对象，用于获取环境、IOC容器、ClassLoader对象等
        2. metadata：注解元对象，获取注解定义的属性值
    2. 判断条件：
      在初始化 Bean 的时候，使用 @Conditional({条件类.class}) 注解

  • Springboot 提供了常用的条件注解

    1. @ConditionalOnClass：判断环境中是否有对应的 class 文件，存在则初始化 Bean
    2. @ConditionalOnProperty： 判断配置文件是否有对应的属性和值，存在则初始化 Bean
    3. @ConditionalOnMissingBean：环境中如果不存在这个 Bean，则初始化

依赖属性配置

业务 Bean 尽量避免设置为强制加载，而是根据需要导入后加载，可以有效降低 Spring 容器管理 Bean 的强度。

将业务功能运行时所需的资源抽取出来独立成类（xxxProperties），读取配置文件信息。

而配置文件中使用固定格式为属性类注入数据。

定义业务功能 Bean，通常使用 @Import 导入，可以有效解耦。

使用 @EnableConfigurationProperties 注解，设置使用属性类时才加载 Bean。
