Bean

Bean 注入流程

  1. 扫描阶段

    • 通过 @ComponentScan 扫描标注 @Component、@Service、@Repository、@Controller 的类。
    • 通过 @SpringBootApplication 自动触发扫描。

  2. 注册阶段

    • 将扫描到的类或通过 @Bean 定义的对象注册到 Spring 容器（BeanDefinition）。
    • 处理 @Import、@Conditional 等注解。

  3. 实例化阶段

    • 根据 BeanDefinition 实例化对象（调用构造函数）。

  4. 属性注入阶段

    • 通过反射注入依赖项（@Autowired、@Resource 等）。

  5. 初始化阶段

    • 调用 @PostConstruct 方法或 InitializingBean.afterPropertiesSet()。

  6. 使用阶段

    • 从容器中获取 Bean 并使用。

  7. 销毁阶段

    • 调用 @PreDestroy 方法或 DisposableBean.destroy()。

注入方法

如果一个接口有多个实现类，则只能用以下这些方式进行注入：

  1. 使用 @Qualifier 指定注入的类名称
  2. 使用 @Primary 标注一个类，注入时默认使用该类
  3. 使用 List<> 或者 Map 遍历接口，获取全部实现类
  4. 使用 @Conditional 根据配置条件进行注入，需要手动额外配置

三、第三方对象的引入方法

1. @Bean 显式注册（推荐）

场景：

  • 需要配置参数（如连接池、超时时间）。
  • 需要绑定外部配置（@Value、@ConfigurationProperties）。

2. @Import 导入（简单对象）

场景：

  • 第三方类有无参构造函数且无需配置。

3. 动态注册（BeanDefinitionRegistry）

场景：

  • 需要根据条件动态注册 Bean。
  • 扩展 Spring 的自动配置（如插件化系统）。

4. XML 配置（遗留系统兼容）

场景：

  • 迁移遗留项目时兼容旧 XML 配置。

四、补充关键问题

1. Bean 作用域控制

场景：

  • 需要多例（非单例）对象时。

2. 条件化注入（@Conditional）

场景：

  • 根据配置文件或环境动态启用/禁用 Bean。

3. 生命周期管理

场景：

  • 需要显式控制初始化/销毁逻辑（如数据库连接池）。

4. 避免循环依赖

场景：

  • 解决 A 依赖 B、B 依赖 A 的循环问题。

5. Spring Boot 3 的 Jakarta EE 9 适配

场景：

  • 升级到 Spring Boot 3 时需处理包名冲突。

五、总结

  • 核心原则：第三方对象必须通过 @Bean 或 @Import 显式注册。
  • 最佳实践：优先使用构造器注入，避免字段注入。
  • 扩展能力：结合 @Conditional 和 @ConfigurationProperties 实现灵活配置。

如需进一步探讨具体场景或代码实现，请随时提问！
