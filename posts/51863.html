SQL 优化的方案有哪些

  1. 索引优化，针对常用的检索条件，例如：where、order、group 后的字段创建索引；
    使用 EXPLAIN 执行计划，确保索引有效命中，减少全表扫描；
    同时也要注意索引变多带来的额外空间成本和写操作的开销
  2. 查询优化，编写查询的时候，尽量选择必要的列，减少回表操作；
    对于分页查询，可以采用深分页或者嵌套子查询的方式，解决深分页问题；
  3. 对表结构进行优化，表设计时可以适当添加冗余字段，方便查询；
    避免使用过大的数据类型，减少存储和查询开销
  4. 减少单个事务的大小和执行时间，采用分批处理的方式，避免长事务
  5. 读写分离
  6. 还无法满足，可以考虑引入缓存，比如redis，减少数据库的直接访问压力

模糊查询左侧加%会失效，该怎样处理

  1. 创建辅助字段，把索引字段使用 REVESER() 反向字符串存储，查询条件也反向传入。
    缺点：空间换时间，额外存储开销，实际使用不多。
  2. 限制查询数量，除了模糊查询字段外，额外添加条件,在业务层面限制查询数量。
  3. 覆盖索引，查询结果中包含主键字段
  4. 使用缓存，减少查询次数

小程序登录逻辑是怎样的

  1. 用户发起登录：调用 wx.login 获取临时 code。
  2. 向后端发送 code：前端将 code 传给后端服务器。
  3. 换取 OpenID 和 SessionKey：后端用 code + appid + appsecret 请求微信接口，获取用户唯一标识 openid 和会话密钥 session_key。
  4. 生成自定义登录态：后端生成 Token（如 JWT），关联用户信息，返回给前端。
  5. 前端存储 Token：前端将 Token 存入本地缓存（如 wx.setStorage）。
  6. 后续请求携带 Token：前端每次请求后端时，携带 Token 进行身份校验。

500w 数据导出为 Excel

  1. 数据库分页或游标逐批读取数据

  2. 需要注意使用的工具是否有限制（EasyExcel 对 Sheet 的数据量有要求，不超过104w）

  3. 分割为多个 Sheet

微信小程序支付流程

佣金分润的功能是在哪一步（支付后？确认收货后？退款了怎么办？）

  1. 创建订单：用户下单，后端生成订单并记录状态（待支付）。
  2. 统一下单接口：后端调用微信支付接口，生成 prepay_id。
  3. 签名返回参数：后端将 prepay_id 等参数签名后返回前端。
  4. 调起支付：前端调用 wx.requestPayment，用户输入密码完成支付。
  5. 支付结果通知：微信异步通知后端支付结果，后端更新订单状态（已支付）。
  6. 佣金分润：会员资格会在支付完成后分润（不支持退款），普通商品在 确认收货后。

Springcloud 用过哪些组件

Springcloud Gateway断言

  1. 作用：定义路由规则，匹配请求是否进入某条路由。
  2. 常用断言：
    • Path：匹配请求路径（如 /api/**）。
    • Query：匹配 URL 参数（如 ?name=xxx）。
    • Header：匹配请求头（如 X-Request-Id）。
    • Method：匹配 HTTP 方法（如 GET、POST）。
    • Cookie：匹配 Cookie 值。
    • Host：匹配域名（如 **.example.com）。

如果不是，则可以成功，因为服务信息此时在缓存中。

Open Feign 底层了解吗

  1. 核心机制：

    • 基于动态代理：为接口生成实现类，将注解转化为 HTTP 请求。
    • 整合 Ribbon：实现负载均衡（从注册中心获取服务列表）。
    • 整合 Hystrix：支持熔断降级（需手动开启）。

  2. 请求流程：

    • 解析接口方法上的 @RequestMapping 注解生成 RequestTemplate。
    • 通过 Encoder 序列化参数，发送 HTTP 请求。
    • 通过 Decoder 反序列化响应结果。

MQ 了解吗

  1. 核心概念：

    • 解耦：生产者与消费者无需直接交互。
    • 异步：生产者发送消息后立即返回，消费者异步处理。
    • 削峰填谷：MQ 作为缓冲区应对流量突增。

  2. 技术选型：

    • RabbitMQ：基于 AMQP，支持复杂路由（Exchange + Binding）。
    • Kafka：高吞吐量，适合日志、大数据场景。
    • RocketMQ：阿里开源，支持事务消息、顺序消息。

  3. 常见问题：

    • 消息丢失：生产者确认机制、MQ 持久化、消费者手动 ACK。
    • 重复消费：业务层幂等性设计。

Myisam 有哪些使用场景

SQL 调优用的是什么

  1. 工具：

    • EXPLAIN：分析执行计划，关注 type（访问类型）、key（使用索引）、rows（扫描行数）。
    • 慢查询日志：定位执行时间过长的 SQL。

  2. 常见手段：

    • 避免 SELECT *，减少数据传输。
    • 用 JOIN 代替子查询。
    • 对 WHERE 条件字段加索引。
    • 分批处理大数据量更新（如 LIMIT 1000）。

常用的设计模式

  • 单例模式 ：确保一个类只有一个实例（如数据库连接池）。
  • 工厂模式 ：解耦对象创建（如根据配置动态生成不同数据库访问类）。
  • 策略模式 ：动态切换算法（如根据不同地区计算运费）。
  • 观察者模式 ：事件驱动（如用户注册后触发发邮件、积分增加）。
  • 装饰器模式 ：动态添加功能（如 Java IO 流的包装）。

Nginx 负载均衡策略有哪些？

  • 轮询（默认）：按顺序分配请求。
  • 加权轮询：根据服务器权重分配流量。
  • IP Hash：同一 IP 的请求固定到同一服务器。
  • 最少链接：优先分配给连接数最少的服务器。
  • URL Hash：按 URL 分配，相同 URL 到同一服务器。

两台服务器，怎样使得更多的请求打到高性能服务器

  • 加权轮询：Nginx 中配置高权重。
  • Least Connections：让处理能力强的服务器快速释放连接，从而接收更多请求。
  • 动态权重：结合监控系统（如 Prometheus），根据实时负载自动调整权重。

接口和实现类的区别

接口描述了某个对象应该具备的行为或能力，但是不关心具体实现，不能直接实例化，一个类可以实现多个接口；

实现类是接口的具体解决方案，为接口中所有抽象方法提供逻辑实现，更关注“如何做”，可以实例化。区别

Spring 的特点和常用注解

简化应用程序的开发和部署，有以下优点：

  1. 对象托管，Spring 能赋值和管理所有对象，不需要手动管理对象的创建和依赖关系

  3. 低侵入性，对于业务代码几乎是无侵入的，只需要使用注解即可
